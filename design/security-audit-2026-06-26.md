### [H1] Master seed sealed under a single-keccak KDF over a 30-bit pairing code — offline brute-force recovers the seed if the QR/URL ciphertext leaks

- **Severity:** high  ·  **Category:** security  ·  **Unit:** crypto-keys  ·  **Confidence:** high  ·  **Verify votes:** 3/3
- **Location:** `src/wallet.rs:304-311`

**What:** adopt_code_key() derives the AES-256-GCM key that seals the MASTER mnemonic for QR device-adoption by hashing a one-time pairing code with a single keccak256 (no salt, no iteration count, no memory-hard KDF): keccak256("localharness/v0/adopt" || uppercase(code)). The code is generated by generate_pair_code() (src/app/events/devices.rs:177-185) as 6 characters over a 32-symbol alphabet = 32^6 = 2^30 possibilities. The resulting ciphertext is produced by encryption::seal_with_raw_key (src/app/encryption.rs:57) and transported BOTH as a QR image and as a copyable URL `https://localharness.xyz/?adopt=1#s=<hex>` (devices.rs:132-133) intended to be moved to a second device. Because the KDF is a single fast hash, an attacker who intercepts only the ciphertext (a photographed/screen-shared QR, the adopt URL pasted into a messaging app, or the fragment left in the receiving device's history) can brute-force all 2^30 candidate codes offline: each trial is one keccak256 + one AES-GCM decrypt, and the GCM auth tag is a perfect success oracle. 2^30 trials is seconds-to-minutes on commodity hardware. Success yields the full BIP-39 master seed = complete identity takeover and loss of all $LH / on-chain assets. The CLAUDE.md documents the adoption FLOW but does not acknowledge or accept this brute-force exposure.

**Evidence:**
```
pub fn adopt_code_key(code: &str) -> [u8; 32] {
    let mut hasher = Keccak256::new();
    hasher.update(b"localharness/v0/adopt");
    hasher.update(code.trim().to_uppercase().as_bytes());
    ... } // single keccak, no salt/stretch
// devices.rs: 6 chars over 32 symbols = 2^30
const ALPHABET: &[u8] = b"23456789ABCDEFGHJKLMNPQRSTUVWXYZ"; let mut bytes = [0u8; 6];
// ciphertext rendered as a shareable URL:
let url = format!("https://localharness.xyz/?adopt=1#s={hex}");
```

**Fix:** Do not protect the master seed with a fast hash of a low-entropy human code. Either (a) replace the code-derived key with the ECIES ephemeral-key handshake already used by seed_pull (no shared low-entropy secret crosses a channel), or (b) if a typed code must stay, use a memory-hard KDF (Argon2id with high memory/time cost) AND raise the code to >=128 bits of effective entropy, AND stop emitting a copy-pasteable URL containing the ciphertext (QR-only, transient). Any change to adopt_code_key needs a versioned migration since it is byte-pinned for CLI parity.

### [H2] MPP on-ramp: USDC.e settlement tx not bound to caller — anyone can claim any treasury-inbound transfer and mint $LH to themselves

- **Severity:** high  ·  **Category:** security  ·  **Unit:** billing-payments  ·  **Confidence:** high  ·  **Verify votes:** 3/3
- **Location:** `C:\Users\kyle\Downloads\apply\antig\proxy\api\mpp-onramp.ts:178`

**What:** The MPP on-ramp mints $LH from a client-supplied settlement tx hash, but never verifies that the on-chain payer of that USDC.e transfer is the authenticated caller. `verifyAuthToken` only gates rate-limit + picks a default recipient; the actual money source is `credential.settlementTx`, and the mint recipient is the server-resolved `payTo` (caller or an arbitrary `body.pay_to`). `_mpp.ts::verifySettlement` even computes `payer` (the on-chain sender) but `mintFromSettlement` ignores it entirely — it mints to `payTo` using only the on-chain amount. Because the one-shot receipt is keyed solely on the tx hash, the FIRST caller to present any unclaimed treasury-inbound USDC.e transfer wins the mint. An attacker who watches the treasury for inbound USDC.e (or finds any abandoned/failed-claim/direct transfer) can call /mpp/onramp with their own valid auth token and the victim's tx hash, crediting the resulting $LH to their own address. The victim then gets an idempotent 'already used' no-op. This is theft of minted platform credit at the victim's expense.

**Evidence:**
```
mpp-onramp.ts: `const payTo = typeof body.pay_to === 'string' && isHexAddress(body.pay_to) ? (body.pay_to as string).toLowerCase() : caller.toLowerCase();` then `const out = await mintFromSettlement(credential.settlementTx, payTo);` — caller is never compared to the settlement's payer. _mpp.ts verifySettlement returns `{ ok: true, lhWei, usdceUnits, payer }` (line ~384) but mintFromSettlement(settlementTx, lhAddress) uses only `v.lhWei` and mints to `lhAddress` (=payTo); `v.payer` is unused. The header comment even reassures only that the recipient isn't 'a free-floating credential field' — it never binds whose payment it is.
```

**Fix:** Bind the proof-of-payment to identity: require `v.payer.toLowerCase() === caller.toLowerCase()` (the authenticated caller must be the on-chain USDC.e sender) before minting, or mint to `v.payer` rather than an arbitrary `payTo`. Keep `pay_to` as an optional credit-direction only AFTER the payer is proven to be the caller.

### [M1] Per-address telemetry rate limit is defeated by free keypair rotation (GitHub-issue spam cannon; can exhaust the shared bot PAT used by publish + scheduler)

- **Severity:** medium  ·  **Category:** security  ·  **Unit:** proxy-authz  ·  **Confidence:** high  ·  **Verify votes:** 3/3
- **Location:** `proxy/api/telemetry.ts:75`

**What:** Auth on /api/telemetry only proves that SOME valid secp256k1 keypair signed the freshness message — `verifyAuthToken` checks signature recovery against the claimed address and a timestamp window, but never checks that the address has any on-chain registration, balance, or session (see _stripe.ts:355-374). Generating unlimited keypairs and signing the token is free and offline. The only abuse control is `senderWindow.hit(addr)` keyed on the authenticated ADDRESS (20/hour, and per-isolate via SlidingWindow). Rotating to a fresh keypair per request bypasses it entirely. Each accepted request files a GitHub issue in the private telemetry repo (the PRIMARY task list) using GH_TELEMETRY_TOKEN. Dedup only suppresses identical `signature` substrings, which the attacker varies. Worse, publish.ts (GH_APPSTORE_TOKEN ?? GH_TELEMETRY_TOKEN) and _jobstore.ts (GH_JOBS_TOKEN ?? GH_TELEMETRY_TOKEN) FALL BACK to the same telemetry PAT 'the moment this ships, before a dedicated token is provisioned' — so flooding telemetry can trip GitHub's secondary rate limit on the shared token and break app publishing AND off-chain scheduling. The header comment's claim 'rate-limited per address so it can't be turned into a spam cannon' is false given free identity minting.

**Evidence:**
```
telemetry.ts header: '...rate-limited per address so it can't be turned into a spam cannon.' Gate: `const wait = senderWindow.hit(addr);` (line 75) where addr is from `verifyAuthToken(token)` (_stripe.ts:355) which only does `recoverAddress(...)` + freshness — no on-chain presence check. Shared token: publish.ts:28 `GH_APPSTORE_TOKEN ?? process.env.GH_TELEMETRY_TOKEN` and _jobstore.ts:37 `GH_JOBS_TOKEN ?? process.env.GH_TELEMETRY_TOKEN`.
```

**Fix:** Add a global/per-IP cap in addition to per-address (even per-isolate helps), and/or require the authed address to have an on-chain identity (e.g. mainOf(addr) != 0 or a minimum $LH balance) before filing. Provision dedicated GH_APPSTORE_TOKEN/GH_JOBS_TOKEN so a telemetry flood cannot rate-limit the publish/schedule paths. Optionally batch/queue issue creation.

### [M2] Open unmetered un-rate-limited authenticated write relays chat and signal exhaust the shared GitHub token and repo causing platform-wide off-chain DoS

- **Severity:** medium  ·  **Category:** security  ·  **Unit:** proxy-injection-dos  ·  **Confidence:** medium  ·  **Verify votes:** 3/3
- **Location:** `proxy/api/chat.ts:99-142, signal.ts 181-300, _jobstore.ts 34-37`

**What:** chat.ts and signal.ts accept POST behind only a personal-sign token, with no meter debit (chat is explicitly an open room) and no rate limit. Neither imports _ratelimit.ts, which exists and is used by notify and broadcast. Each POST performs one or two GitHub Contents-API writes against a single repo, compusophy/localharness-jobs, the default for CHAT_REPO, SIGNAL_REPO and JOBS_REPO, using a single PAT. Personal-sign tokens are free and offline to mint from unlimited throwaway wallets, so the auth requirement is not a throttle. Sustained spam exhausts the token GitHub primary and secondary rate limits, which also feed the scheduler cron claim and advance, apps discovery and signaling, taking the whole off-chain platform offline, and floods the repo with commits and new files, risking GitHub abuse-suspension of the token. Per-owner job caps are bypassed with fresh wallets, and chat and signal have no cap at all.

**Evidence:**
```
chat.ts handler calls ghWrite after only verifyAuthToken with no SlidingWindow or meter, and signal.ts join, post and put-slots call ghPut after only verifyAuthToken. _jobstore.ts lines 34-37 set GH_TOKEN to GH_JOBS_TOKEN else GH_TELEMETRY_TOKEN, and chat.ts 16-17 plus signal.ts 27-28 default to the same repo and token. _ratelimit.ts is not imported by chat, signal or schedule.
```

**Fix:** Apply a SlidingWindow rate limit per claimed address and per source IP on chat and signal POST, plus a global write budget. Isolate signal and chat into their own repo and token so abuse cannot starve the scheduler cron. Add a small meter debit or a proof-of-membership on POST, and consider a TTL KV as the module headers anticipate.

### [M3] _jobstore countByOwner and listByOwner read every job body on each create, an O of total jobs GitHub-read amplification

- **Severity:** medium  ·  **Category:** security  ·  **Unit:** proxy-injection-dos  ·  **Confidence:** high  ·  **Verify votes:** 3/3
- **Location:** `proxy/api/_jobstore.ts:287-305, called from schedule.ts 180`

**What:** Because the job filename encodes only nextRun and id, not owner, countByOwner cannot filter from the listing alone. listByOwner lists the directory then performs a GitHub content GET for every job body, up to the 1000-entry listing cap, to find the ones matching one owner. schedule.ts calls countByOwner on every create attempt, before the per-owner cap is even confirmed. So a single create request fans out to potentially hundreds or thousands of upstream GitHub reads, and an attacker spamming create, even attempts that ultimately fail the cap, multiplies consumption of the shared GitHub token, directly amplifying the chat and signal platform DoS and slowing every legitimate create and cron tick. The per-owner cap does not bound the global store since an attacker uses many wallets, and the 1000-entry listing cap means a flooded jobs directory can crowd legitimate due jobs out of the cron scan.

**Evidence:**
```
listByOwner lists entries then for each entry reads the body via ghReadBody and pushes it when read.job.owner lowercased equals owner lowercased. countByOwner returns the length of that list.
```

**Fix:** Encode the owner, or a short prefix of it, into the job filename so countByOwner resolves from the directory listing with zero body reads, the same trick already used for nextRun. Add a global active-job cap and rate-limit or meter create.

### [M4] Spend-to-zero min(cost,avail) + per-model tiered pricing lets a dust balance buy a premium-tier (e.g. 20 $LH Opus) call

- **Severity:** medium  ·  **Category:** security  ·  **Unit:** billing-payments  ·  **Confidence:** medium  ·  **Verify votes:** 3/3
- **Location:** `C:\Users\kyle\Downloads\apply\antig\proxy\api\gemini.ts:655`

**What:** The credit gate serves any request when `availCredit > 0` and charges `min(cost, avail)`. With Gemini's flat 1 $LH this is the documented 'leftover buys a message' behavior, but Anthropic/OpenAI pricing is tiered up to 20 $LH (Opus). So a caller holding any dust balance — even 0.01 $LH — can invoke a 20 $LH Opus request and be charged only the dust, with the ~19.99 $LH of provider cost absorbed by the platform. Single-account this is one underpriced call per balance-exhaustion, but because identities register for free and $LH transfers are sponsored, a sybil can fan out a small purchase across many addresses each holding dust and convert each into a full premium-tier call, amplifying provider-dollar cost well beyond the $LH actually spent. The documented justification ('don't strand a leftover < one message') was written for the flat-price case and does not account for a 20x tier.

**Evidence:**
```
gemini.ts: `const meterCharge = availCredit < cost ? availCredit : cost; // min(cost, avail)` and `const hasCredit = availCredit > 0n;` then the gate `if (!hasSession && !hasCredit && !paidViaX402) return 402`. _prices.ts: PRICE_ANTHROPIC `'claude-opus-4-8': ...20_000_000_000_000_000_000n // 20 $LH` vs default Gemini COST_PER_REQUEST_WEI = 1 $LH.
```

**Fix:** Require `availCredit >= cost` for premium tiers (only allow partial spend-down at or near the flat base tier), or cap the per-call platform subsidy (e.g. serve a premium model only if avail covers some minimum fraction of its list price). Re-evaluate the 'spend to zero' rule now that pricing is per-model tiered rather than flat.

### [M5] Cross-origin signer's $LH allowlist misses transferWithMemo/transferFromWithMemo, bypassing the #81 owner-check and the confirm-gate

- **Severity:** medium  ·  **Category:** security  ·  **Unit:** client-trust  ·  **Confidence:** high  ·  **Verify votes:** 3/3
- **Location:** `src/app/signer.rs:564-588`

**What:** build_sponsored_tx_response is the documented trust boundary for cross-origin tx signing: for a call to the $LH token it enforces a per-selector policy (transferFrom rejected, approve must target the diamond, transfer requires the requesting subdomain be one the master OWNS on-chain via needs_owner_check). The #81 comment states this guarantees a value-moving $LH transfer is signed ONLY for a subdomain the master owns, closing the confused-deputy where a hostile *.localharness.xyz the victim merely VISITS makes the master sign transfer(attacker, balance). But the policy keys on three exact 4-byte selectors and routes everything else to `_ => {}` ('Other selectors move no $LH — allowed'). The $LH token (LocalharnessCredits.sol:190/196) exposes transferWithMemo(address,uint256,bytes32) and transferFromWithMemo(...), both of which call _transfer(msg.sender,to,amount) and move value, yet neither selector is gated. A hostile trusted subdomain can therefore embed the apex signer iframe, post lh-sign-digest with a token call to transferWithMemo(attacker, master_balance, 0), and obtain a master-wallet-signed sponsored tx WITHOUT the ownership check and WITHOUT the chat-layer confirm gate (which only guards the send_lh tool path, not direct iframe requests). The signer auto-signs allowlisted calls with no user consent dialog (despite the doc comment claiming one), so this is silent. Impact: on testnet the embedded sponsor in sponsor.rs signs the fee_payer half locally with no allowlist, so the attacker can assemble and broadcast the tx and drain the victim's testnet $LH. On mainnet the relay (proxy/api/sponsor.ts:399-417 + :507) refuses any token call other than transfer/approve and requires caller==sender with a personal-sign the attacker cannot forge, so mainnet broadcast is currently blocked — but the signer's own control, which is documented as THE protection, is defeated.

**Evidence:**
```
signer.rs token policy: `Some([0xa9,0x05,0x9c,0xbb]) => { needs_owner_check = true; }` ... `// Other selectors move no $LH (e.g. accidental no-ops) — allowed.\n_ => {}`. The asserted '_ => {} moves no $LH' is false: LocalharnessCredits.sol:190 `function transferWithMemo(address to, uint256 amount, bytes32 memo) external returns (bool) { _transfer(msg.sender, to, amount); ... }` moves value and has a selector not in {0x23b872dd, 0x095ea7b3, 0xa9059cbb}.
```

**Fix:** Make the token-call policy default-DENY: reject any $LH-token call whose selector is not explicitly allowed, and add transferWithMemo (treat exactly like transfer → needs_owner_check) and transferFromWithMemo (reject, like transferFrom) to the match. Do not rely on the relay as the only enforcement; the cross-origin signer must be self-contained because the embedded testnet sponsor co-signs anything.

### [M6] RootedFilesystem sandbox escape on Windows via backslash path components

- **Severity:** medium  ·  **Category:** security  ·  **Unit:** sandbox  ·  **Confidence:** high  ·  **Verify votes:** 3/3
- **Location:** `src/filesystem/rooted.rs:34-40`

**What:** RootedFilesystem is the documented bashlite confinement primitive ('a bashlite script can't escape its root'). Confinement relies entirely on the caller handing it already-normalized `/`-rooted paths, but `real()` only concatenates `base + p` and never normalizes `p`. bashlite's `builtins::resolve` (src/bashlite/builtins.rs:414) splits and collapses `..` ONLY on the `/` separator, so a path component containing backslashes is treated as a single opaque literal and passes through verbatim. The bashlite lexer preserves literal backslashes inside single quotes (src/bashlite/lexer.rs:112-125) and double quotes (144), so a script word like '..\..\..\secret' survives expansion intact. On Windows, NativeFilesystem resolves the path via PathBuf::from(path) (src/filesystem/native.rs:56) and std::path treats `\` as a separator, so the `..` segments climb above `base`. Example: `cat '..\..\..\..\Users\kyle\.lh_claude_mainnet.key'` from a script rooted in a project subdir reads the mainnet money key in $HOME. This yields arbitrary host file read/write/delete (via cat/write/mkdir/rename/delete builtins) outside the sandbox, including the on-chain signing keys the CLI stores in $HOME. The browser `execute_script` path runs over OPFS where `\` is not a separator, so the escape is native/Windows-specific.

**Evidence:**
```
rooted.rs real(): `let p = if p.starts_with('/') { p.to_string() } else { format!("/{p}") }; format!("{}{}", self.base, p)` — no `\`→`/` normalization and no `..` rejection on the incoming path. builtins.rs resolve(): `for comp in combined.split('/') { match comp { "" | "." => {}, ".." => { stack.pop(); }, c => stack.push(c) } }` — only `/` is a separator, so `..\foo` is one pushed literal. lexer.rs single-quote arm pushes every byte (incl. `\`) literally.
```

**Fix:** Make RootedFilesystem self-defending: in `real()`, normalize separators (`p.replace('\\', "/")`), reject or collapse any `..` component AFTER the join, and verify the resulting path still has `base` as a prefix before passing it to the inner fs (defense-in-depth, since RootedFilesystem is the security boundary). Additionally, have `builtins::resolve` treat `\` as a separator (or reject it) so cwd/path normalization is platform-correct.

### [M7] bashlite dry-run-manifest confirm gate bypassable via cross-pass divergence

- **Severity:** medium  ·  **Category:** security  ·  **Unit:** sandbox  ·  **Confidence:** high  ·  **Verify votes:** 3/3
- **Location:** `src/bin/localharness/sh.rs:153-209`

**What:** The dry-run-manifest gate is meant to enforce that 'a script can't silently spend/transfer': pass 1 runs the whole script with dry_run=true to collect a value-move manifest shown to the user, and pass 2 re-runs it live only after --confirm. The flaw is that the gate assumes the two runs are identical, but (a) the two passes execute the SAME script over the SAME on-disk base directory (run_pass builds a fresh RootedFilesystem over NativeFilesystem at `base` each time, sh.rs:99) and (b) the fs builtins `write`/`create`/`mkdir`/`rename`/`delete` are NOT gated by dry_run — only the lh-* value moves are (CliBashHost.run_builtin step 3, sh.rs:83-84, calls builtins::dispatch_in unconditionally). So a script can leave persistent on-disk state during the dry pass and branch on it during the live pass. Concretely: dry pass checks a marker file (absent), plans a tiny benign `lh-send 0.001 $LH -> self` (so the manifest is non-empty and a benign plan is shown), and creates the marker; the user sees the benign plan and runs --confirm; the live pass observes the marker and instead executes `lh-send 1000 $LH -> attacker`. The confirmed plan is not binding on what executes, defeating the gate's purpose.

**Evidence:**
```
sh.rs run_source: pass 1 `run_pass(..., true)` then, only on --confirm, pass 2 `run_pass(src, base, identity, signer, sponsor, fee_token, false)` — the same `src`/`base` re-run. builtins.rs write_create writes via `fs.write_atomic(&p, ...)` with no dry-run awareness. platform.rs lh_send: in dry-run returns the plan string and sends nothing; live it calls `transfer_lh_sponsored(...)` — the live recompute is unrelated to the manifest the user approved.
```

**Fix:** Do not re-execute the script to apply moves. Either (i) execute exactly the move list captured in the dry-run manifest (bind execution to the approved plan, not a re-run), or (ii) gate ALL state-changing builtins (fs writes included) behind dry_run so the dry pass has zero side effects, and additionally hash/snapshot the script + relevant inputs so a divergent second pass is detected and aborted.

### [M8] Subagent filesystem tools bypass the parent's tool policies/hooks (workspace_only escape)

- **Severity:** medium  ·  **Category:** security  ·  **Unit:** builtins  ·  **Confidence:** high  ·  **Verify votes:** 3/3
- **Location:** `src/builtins/start_subagent.rs:351`

**What:** start_subagent builds a subagent with the full 8 filesystem builtins over the SAME Filesystem the parent uses, but dispatches every subagent tool call with hook_runner = None: `dispatch_tool_call(Some(&runner), None, &turn_ctx, &tool_call)`. In dispatch.rs (lines 81-85) a None hook_runner yields `HookResult::allow()` unconditionally — no PreToolCall hooks run. The parent's confinement (`policy::workspace_only`, any `Policy::deny`, ask-user/audit hooks) is implemented as a PreToolCallDecideHook in the HookRunner, so it is enforced for the PARENT's fs calls but NOT for the subagent's. A native SDK consumer who confines an agent to a directory via workspace_only and also grants start_subagent (a plausible 'delegate work' setup) is fully escaped: the model calls start_subagent, and the subagent can view/create/edit/delete/rename/search ANY path on disk (only PROTECTED_FILES basenames are still blocked, since that check lives inside each tool). This directly contradicts the module's documented invariant that 'a subagent's tool surface is a strict reduction of the parent's' (start_subagent.rs:14-16, 388) — in policy terms it is a SUPERSET, since the parent's fs writes are gated and the subagent's are not.

**Evidence:**
```
// No hooks/policies for the detached subagent: pass None so the
// pipeline runs execute + error-lift with the same semantics.
let result = dispatch_tool_call(Some(&runner), None, &turn_ctx, &tool_call).await;

// dispatch.rs 81-85:
let (decision, op_ctx) = if let Some(hooks) = hook_runner { ... } else { (HookResult::allow(), turn_ctx.clone()) };
```

**Fix:** Thread the parent's hook_runner into StartSubagent (store it at construction, like the filesystem) and pass it to dispatch_tool_call so the subagent inherits at least the parent's PreToolCall deny/containment policies. If full hook inheritance is undesirable, at minimum re-apply a path-containment policy to the subagent's runner, or document loudly that start_subagent escapes policy-based confinement and gate it behind the same workspace the parent uses (e.g. hand the subagent a RootedFilesystem instead of the shared one).

### [M10] Off-chain agent jobs are claimed and consumed with zero model calls under wall-clock pressure — one-shot agent jobs silently deleted without ever running

- **Severity:** medium  ·  **Category:** correctness  ·  **Unit:** proxy-correctness  ·  **Confidence:** high  ·  **Verify votes:** 1/1
- **Location:** `proxy/api/scheduler.ts:2196 (deadline calc), 2037 (claim), 2134 (writeNextSlot), 1155 (clock cap)`

**What:** fireOffchainDue computes each off-chain agent's fair-share model deadline as `tickStart + TICK_SOFT_BUDGET_MS*(i+1)/agents.length` — anchored to tickStart, IGNORING the wall-clock the on-chain batch (and reminders) already consumed before off-chain agents start. The only guard before starting agent i is `Date.now() - tickStart >= TICK_SOFT_BUDGET_MS` (full budget gone). So when the on-chain batch has eaten, say, 15s of a 20s budget, off-chain agent 0's deadline (e.g. tickStart+5s for N=4 agents) is ALREADY IN THE PAST. fireOffchainJob then CLAIMS the job (claimJob deletes the file, line 2037) BEFORE runPingPong's clock check runs; runPingPong immediately clock-caps at round 0 (mayMeterCall, line 1155) and returns calls=0. Back in fireOffchainJob (no throw, ran='ok', spentWei=0, debit skipped) writeNextSlot is still called (line 2134), CONSUMING the fire (runsLeft-1). For a one-shot agent job (runs:1) writeNextSlot returns null and the file stays deleted — the scheduled agent task is GONE, having never run the model. The on-chain path avoids this via the mid-batch defer guard (line 2349 defers i>=1 without writing) and tickStart≈0 for job 0; the off-chain agent path has no equivalent 'spill instead of claim when my deadline already passed' check.

**Evidence:**
```
Line 2196: `const modelDeadline = tickStart + Math.floor((TICK_SOFT_BUDGET_MS * (i + 1)) / agents.length);` — anchored to tickStart over ALL agents, not remaining-budget over remaining agents. Line 2191 guard only checks the GLOBAL budget: `if (Date.now() - tickStart >= TICK_SOFT_BUDGET_MS) { ...break; }`. ### [L19] host_net SSRF allow-list under-blocks loopback (hex-encoded dotted IPv4 + trailing-dot localhost)
```

- **Severity:** low  ·  **Category:** security  ·  **Unit:** sandbox  ·  **Confidence:** medium  ·  **Verify votes:** 3/3
- **Location:** `src/rustlite/loader.rs:458-485`

**What:** `url_is_allowed` is the SSRF gate for cartridge-opened WebSockets; it is reachable in the browser via the `compile_rustlite` tool, which instantiates the compiled cartridge and calls an exported function (src/builtins/compile_rustlite.rs:105-120 → Cartridge::load/call_i32), and the same logic is hand-ported to web/cartridge-worker.js for the live display runtime (untrusted, published/on-chain cartridges run in a visitor's tab). The gate's IPv4-literal block only rejects a dotted quad whose octets are ALL ascii digits, and only checks the literal strings localhost/*.localhost/*.local. **Evidence:**
```

```

**Fix:** Normalize the host before classification (trim a single trailing dot, lowercase) and reject any host that the WHATWG IPv4 parser would interpret as an address: parse each dotted label with the same hex(0x)/octal(0 prefix)/decimal rules the browser uses and reject if it forms an IPv4, and reject single-number hosts. ### [I1] EncryptedFilesystem.read ignores EXEMPT_FILES — write/read asymmetry in the exemption invariant

- **Severity:** info  ·  **Category:** cleanup  ·  **Unit:** crypto-keys  ·  **Confidence:** high  ·  **Verify votes:** 1/1
- **Location:** `src/filesystem/encrypted.rs:172-188`

**What:** write_atomic() short-circuits on is_exempt(path) so exempt files are stored verbatim, but read() never consults is_exempt — it only sniffs the LHE1 magic via looks_sealed(). For exempt files whose stored bytes are themselves a ciphertext (e.g. .lh_device_key, which key_store/persist_device_key encrypts with the per-origin device key, producing a blob beginning with a random 12-byte IV), there is a ~2^-32 chance the blob's first 4 bytes equal 'LHE1' and length >= 32, in which case read() would route the exempt file through the seed-key GCM decrypt, fail auth, and return an error (manifesting as 'device key lost' → a new key is generated). Impact is negligible (only .lh_device_key is at risk among exempt files, the odds are 2^-32, and the failure is recoverable), but the asymmetry means the EXEMPT invariant is enforced on write only, not on read — a latent footgun if a future exempt file legitimately starts with the magic.

**Evidence:**
```

    let bytes = self.inner.read(path).await?;

}
async fn write_atomic(...) { if Self::is_exempt(path) { return self.inner.write_atomic(path, bytes).await; } ... }
```

**Fix:** Mirror the write guard on read: `if Self::is_exempt(path) { return self.inner.read(path).await; }` before the looks_sealed sniff, so exempt files always pass through verbatim in both directions.
