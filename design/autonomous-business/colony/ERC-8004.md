# ERC-8004 — verifiable agent identity + cross-agent trust for the colony

> How colony agents prove who they are and trust each other + outside agents,
> mapped onto what localharness already ships. Research current to 2026-06.
> Verified live against Tempo mainnet (chain 4217) on 2026-06-30 — addresses
> below are read from the diamond, not copied from a stale doc.

## 1. What ERC-8004 is (current state)

**ERC-8004 "Trustless Agents"** is the Ethereum standard for giving autonomous
AI agents a portable on-chain identity plus standardized trust signals, so agents
can discover, authenticate and transact across organizational boundaries with no
central intermediary. It is explicitly a **trustless extension of Google's A2A
(Agent-to-Agent) protocol**: A2A says how agents *talk*; ERC-8004 adds *who they
are* and *whether to trust them*.

It went **live on Ethereum mainnet (late Jan / Feb 2026)** and is now deployed at
**the same addresses across 40+ chains** (Ethereum, Base, Polygon, Arbitrum,
Optimism, Avalanche, BSC, Linea, Scroll, Celo, Mantle, …). Still a **draft EIP**
under active revision (the Validation registry especially). **Tempo is NOT in the
canonical 40-chain set** — relevant to interop, see §4.

Three lightweight registries:

| Registry | Purpose | Key functions (canonical 0x8004…) |
|---|---|---|
| **IdentityRegistry** (ERC-721) | Each agent = an NFT `agentId` = a portable passport. Token URI → a registration JSON (the agent card). | `register(string agentURI) -> agentId`, `setAgentURI`, `setMetadata/getMetadata(agentId,key)`, `setAgentWallet(agentId,wallet,sig)` / `getAgentWallet` (EIP-712 / ERC-1271 proof of the receiving wallet) |
| **ReputationRegistry** | Client feedback as a signed decimal (`value`+`valueDecimals`) with tags + an optional `feedbackURI`. Aggregates to a summary. | `giveFeedback(agentId, value, valueDecimals, tag1, tag2, feedbackURI, feedbackHash)`, `readFeedback`, `readAllFeedback`, `getSummary -> (count, summaryValue, decimals)`, `revokeFeedback`, `appendResponse` |
| **ValidationRegistry** (evolving) | Standardizes "proof a task was done" — request/response hooks for zkML proofs, TEE oracles, or **staked re-execution**. | `validationRequest(validator, agentId, requestURI, requestHash)`, `validationResponse(requestHash, response, responseURI, responseHash, tag)`, `getValidationStatus`, `getSummary`, `getAgentValidations`, `getValidatorRequests` |

**Canonical live addresses** (same on every supported chain):
- Mainnet IdentityRegistry `0x8004A169FB4a3325136EB29fA0ceB6D2e539a432`,
  ReputationRegistry `0x8004BAa17C55a88189AE136b182e5fdA19dE9b63`.
- Testnet (Sepolia/Amoy/…) IdentityRegistry `0x8004A818BFB912233c491871b3d84c89A494BD9e`,
  ReputationRegistry `0x8004B663056A597Dffe9eCcC1965A193B7388713`.

**Agent identifier schema:** an agent is `(agentRegistry, agentId)` where
`agentRegistry = "{namespace}:{chainId}:{identityRegistry}"` (CAIP-style, e.g.
`eip155:1:0x8004A169…`) and `agentId` is the ERC-721 `tokenId`.

## 2. The agent card / `.well-known/agent.json` convention (+ A2A)

ERC-8004's registration file **is an A2A Agent Card with extra on-chain fields.**
The `agentURI` (the ERC-721 token URI) resolves to JSON; an agent that advertises
an HTTPS endpoint can *prove domain control* by serving the same document at
**`https://<domain>/.well-known/agent.json`** (A2A v0.3 renamed the canonical file
to `agent-card.json`; both names are seen in the wild — serve both).

Shape (A2A core ∪ ERC-8004 additions):

```jsonc
{
  // --- A2A core ---
  "protocolVersion": "0.3.0",
  "name": "...", "description": "...", "url": "https://…",
  "version": "...", "provider": { "organization": "...", "url": "..." },
  "capabilities": { "streaming": true, "pushNotifications": true },
  "skills": [ { "id", "name", "description", "tags", "examples" } ],
  "securitySchemes": { ... },        // OpenAPI-style auth declarations
  // --- ERC-8004 additions ---
  "type": "<registration schema id>",
  "services": [ { "type": "a2a"|"mcp"|"https", "url": "..." } ],
  "registrations": [ { "agentRegistry": "eip155:…:0x…", "agentId": "…" } ],
  "supportedTrust": ["reputation", "crypto-economic", "tee-attestation"]
}
```

**Three trust models** an agent advertises in `supportedTrust`:
- `reputation` — client feedback via the ReputationRegistry.
- `crypto-economic` — staked validation / re-execution (skin in the game).
- `tee-attestation` — hardware attestation (TEE) / zk proof of correct execution.

A2A's contribution: `capabilities`, `skills`, `securitySchemes`, streaming /
push semantics — capability negotiation **without** exposing internal state.
ERC-8004's contribution: the cryptographic identity (`registrations`), the
receiving wallet (`setAgentWallet`), and the on-chain trust legs.

## 3. Fit vs what localharness already ships

localharness already has the **whole shape** of ERC-8004 — under different
function names — cut into ONE EIP-2535 diamond on Tempo. Verified live on mainnet
(chain 4217, diamond `0x8ab4f3a57643410cdf4022cdaf1faeef234f3a77`, 36 facets):

| ERC-8004 leg | localharness equivalent | Live on mainnet? |
|---|---|---|
| IdentityRegistry (ERC-721 `agentId`) | `LocalharnessRegistryFacet` + `ERC721Facet`: every name is an NFT at `<name>.localharness.xyz`; `idOfName(string)`/`ownerOf`/`tokenURI`. | ✅ yes |
| Agent wallet (`setAgentWallet`/`getAgentWallet`) | `TbaFacet`: every name has an **ERC-6551** token-bound account (the agent's wallet) — `tokenBoundAccount(id)`. Stronger than a settable pointer: it's a contract account that travels with the NFT. | ✅ yes |
| Registration file / token URI | `setMetadata`/`metadata(id,key)` (persona, public_face, x402_price); `tokenURI(id) -> https://<name>.localharness.xyz/`. | ✅ yes |
| ReputationRegistry | `ReputationFacet`: `attest(subjectTokenId, 1..5, workRef)` + `reputationOf -> (count, ratingSum)` + `attestationsOf`. The colony engine auto-attests the worker after a paid cycle. | ✅ **cut** (impl `0x8D83348B6eedB6903aa6735D57A99b1187A3ca16`; 0 attestations so far) |

### The gap (what is NOT ERC-8004-compliant yet)

1. **Function-name / ABI mismatch.** Our identity surface is `idOfName` /
   `register(name)` / `setMetadata`, not ERC-8004's `register(agentURI)` /
   `setAgentURI` / `getMetadata(agentId,key)` / `setAgentWallet`. A generic
   ERC-8004 indexer querying our diamond by the canonical selectors gets nothing.
2. **No registration JSON was served.** We had persona/face metadata but no
   `.well-known/agent.json` / token-URI document in the ERC-8004 + A2A shape, so
   nothing advertised `registrations` / `supportedTrust` / the agent wallet to
   other agents. **(Closed by this change — §6.)**
3. **Not on a canonical 0x8004 registry.** Tempo isn't one of the 40+ chains where
   `0x8004A169…` is deployed, so an ERC-8004-native agent on Base/Ethereum can't
   discover a localharness agent by reading the canonical IdentityRegistry.
4. **Reputation semantics differ.** Ours is a 1..5 attestation by an identity;
   ERC-8004's is a signed-decimal client feedback with tags + `feedbackURI`. Close
   in spirit, not wire-compatible.

## 4. The interop plan

Two complementary levels — do the free one now, gate the on-chain one on the key.

### Level A — native, self-describing cards (autonomous, DONE)

Treat each localharness diamond as an **ERC-8004-compatible identity registry in
its own right** and publish A2A+ERC-8004 cards that point at it:

- Serve `/.well-known/agent.json` (+ `agent-card.json`) per agent describing
  `name`, `description` (persona), `skills`, `capabilities`, the on-chain
  `registrations` (`eip155:4217:0x8ab4…:agentId`), the ERC-6551 `agentAddress`
  (the receiving wallet), and `supportedTrust` derived from which facets are cut.
- **Map the ValidationFacet to the `crypto-economic` leg** and ReputationFacet to
  the `reputation` leg directly in the card's `trustModels`, with the live facet
  impl addresses (verifiable via DiamondLoupe) and the exact method signatures.
- Advertise the **x402 payment** model: pay-to = the agent's TBA, asset = `$LH`,
  price = `x402_price` (default 0.01), domain separator read live.

A consumer that speaks A2A can already negotiate capabilities; a consumer that
speaks ERC-8004 gets the registry pointer, the wallet, and the trust legs — all
verifiable on Tempo. This is **full ERC-8004 *card* compatibility** without the
canonical-registry deployment.

Serving model (two tiers, both built here):
- **Static (flagship, web deploy):** `web/.well-known/agent.json`. The apex +
  every subdomain share one wildcard web project, so a static file is the SAME on
  all hosts — correct for the flagship `localharness` agent; a stopgap elsewhere.
- **Dynamic (per colony agent, proxy deploy):** `proxy/api/agent-card.ts` builds
  the card LIVE from chain for any `?name=` (or Host subdomain). Activate by
  adding a host-keyed rewrite to the web `vercel.json` so
  `<name>.localharness.xyz/.well-known/agent.json` routes to the proxy:
  ```jsonc
  { "source": "/.well-known/agent.json",
    "has": [{ "type": "host", "value": "(?<name>.*)\\.localharness\\.xyz" }],
    "destination": "https://proxy-tau-ten-15.vercel.app/api/agent-card?name=:name" }
  ```

### Level B — canonical 0x8004 cross-chain registration (needs the owner's key)

To be discoverable by ERC-8004-native agents on Ethereum/Base/etc., a localharness
agent also registers on the canonical IdentityRegistry on a supported chain:

1. On (say) Base, `IdentityRegistry(0x8004A169…).register(agentURI)` where
   `agentURI = https://<name>.localharness.xyz/.well-known/agent.json`.
2. `setAgentWallet(agentId, <the agent's address>, sig)` to bind the receiving
   wallet (EIP-712 / ERC-1271 — the ERC-6551 TBA can sign via 1271).
3. The card's `registrations` array now lists BOTH the Tempo diamond and the
   canonical Base registry, cross-linking the identity.

This is a **mainnet write on another chain → it needs the owner's key and gas on
that chain. Custody stays human; not autonomously executable.** (Optional deeper
work: an `ERC8004AdapterFacet` exposing `register`/`setAgentURI`/`getMetadata`/
`setAgentWallet` selectors over our existing name-NFT storage, so the localharness
diamond answers canonical ERC-8004 calls directly — a facet cut, also key-gated.)

## 5. Buildable NOW vs needs a funded key

**Autonomous now (read-only + artifacts — done in §6):**
- The agent-card JSON in ERC-8004 + A2A shape, accurate to live chain state.
- Static serving (`web/.well-known/agent.json` + `agent-card.json` alias + headers).
- The dynamic per-agent endpoint (`proxy/api/agent-card.ts`), tsc-clean.
- Reading identity/reputation/validation/x402 state to populate cards.

**Needs the owner's key / a deploy (custody stays human):**
- Deploying the artifacts (`vercel deploy --prod` for web; `cd proxy && vercel
  --prod` + the host-rewrite for dynamic cards). No key, but a deliberate deploy.
- Level-B canonical `0x8004` registration on Base/Ethereum (a tx on that chain).
- Binding the agent wallet there (`setAgentWallet`, EIP-712/1271 signature).
- Any on-chain attest / stakeValidation to seed real reputation/validation.
- An `ERC8004AdapterFacet` `diamondCut` (owner cut key).

## 6. What this change builds

- **`web/.well-known/agent.json`** — the real, live-accurate agent card for the
  flagship `localharness` agent (agentId 9): persona, skills, on-chain
  registration, ERC-6551 wallet, `supportedTrust: [reputation, crypto-economic]`
  each mapped to its live facet, and the x402 payment model. Served at the
  apex/flagship on a web deploy.
- **`web/.well-known/agent-card.json`** — A2A v0.3 canonical filename; served as
  the same document via a `vercel.json` rewrite.
- **`vercel.json`** — a `/.well-known/*.json` headers rule (`application/json` +
  open CORS so any agent can fetch cross-origin) + the agent-card alias rewrite.
- **`proxy/api/agent-card.ts`** — dynamic per-agent card endpoint (Level A,
  per-subdomain). Reads `idOfName`/`ownerOf`/`tokenBoundAccount`/`metadata` and
  DiamondLoupe `facetAddress` to report only the trust legs actually cut. Built +
  type-checked; **inert until `cd proxy && vercel --prod` + the host-rewrite.**

Nothing here fires a transaction; every on-chain value is read-only.
